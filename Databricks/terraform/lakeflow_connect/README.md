# lakeflow_connect — per-domain CDC ingestion (Terraform + DAB)

Ingest data from external **source systems** ("domains" — HR, Finance,
Business Development, …) into Databricks Unity Catalog bronze Delta tables. The pipeline is
**configuration-driven per domain** and split across two tools by what each does
best:

- **Terraform (`ingestion/`)** — per-domain Unity Catalog governance: storage
  credential, external location, optional Lakehouse Federation. One module
  instance per domain via `for_each = var.domains`.
- **Databricks Asset Bundle (`bundles/lakeflow_connect`)** — builds the
  `cdc_bronze_ingest` wheel and deploys each domain's
  UC assets and Auto Loader ingestion job.

A "domain" is an **external source system, not a cloud stack**. Each domain's
landing infrastructure (e.g. an S3 bucket + IAM role) is assumed to already exist; the
[`aws/`](aws/) directory is a **single worked example** showing how to stand one
up end-to-end (RDS → DMS → S3 + IAM).

> The goal was [Databricks Lakeflow
> Connect](https://docs.databricks.com/aws/en/ingestion/lakeflow-connect/), but
> managed MySQL connectors are preview/non-free-tier, so this lands source data
> as files in object storage and ingests with Auto Loader.

## Architecture

```
external system (domain)
    └── lands files in object storage (S3 today; abfss/gs pluggable)
          └── Databricks Auto Loader (cdc_bronze_ingest wheel)
                └── <domain>_bronze Delta tables (Unity Catalog)

Stream state (checkpoints + inferred schema) lives in a UC managed volume under
the bronze schema — not in the source bucket — so the source can be read-only.
Queryable DB domains also get an optional Lakehouse Federation foreign catalog.
```

## Layout

```
lakeflow_connect/
├── aws/              — a worked example: RDS/DMS/S3/IAM for a domain (pure AWS)
├                       for testing purposes.
├── ingestion/        — modular per-domain UC governance (Terraform root)
│   └── terraform.tfvars   — the domains map (single source of truth)
├── modules/
│   └── domain_ingest/ — per-domain module: storage credential + external location
│                        + optional federation; generates the DAB job/schema file
└── bundles/
    └── lakeflow_connect/   — the Databricks Asset Bundle
        ├── cdc_bronze_ingest/   — the Auto Loader ingestion wheel (built here)
        └── resources/           — per-domain *.gen.yml (generated by Terraform)
```

Each sub-tree has its own README: [`ingestion/`](ingestion/README.md),
[`modules/domain_ingest/`](modules/domain_ingest/README.md),
[`bundles/lakeflow_connect/`](bundles/lakeflow_connect/README.md), [`aws/`](aws/README.md).

Ownership: **Terraform** owns UC governance (credential, location, federation);
the **DAB** owns the wheel build, bronze schema, checkpoint volume, and ingestion
job; **pre-existing** per domain is the bucket + IAM role.

## Cloud-pluggable

Each domain declares a `source_infrastructure` with a `type` discriminator. Only
`type = "aws"` is implemented today; adding `azure`/`gcp` later means one new
credential auth block + one storage-URI map entry in `modules/domain_ingest` — the
domain interface and the DAB stay unchanged.

## Deploy

```bash
# 0. (once per new domain) stand up its bucket + IAM role — see aws/ for the example

# 1. Terraform: per-domain UC governance + generate the bundle's resource files
cd ingestion
cp terraform.tfvars.example terraform.tfvars   # edit the domains map
terraform init
terraform apply

# 2. Bundle: build the wheel, deploy each domain's schema + volume + job
cd ../bundles/lakeflow_connect
databricks bundle validate
databricks bundle deploy -t dev

# 3. Run a domain's ingestion
databricks bundle run hr_cdc_ingest -t dev
```

Apply order matters: Terraform generates `resources/*.gen.yml` that the bundle
deploys, so **Terraform first, then the bundle**.

## Adding a domain

Add an entry to `domains` in `ingestion/terraform.tfvars` (with the domain's
pre-existing `role_arn`/`bucket`/`prefix`, and a `federation` block only for
queryable DBs), then `terraform apply` + `databricks bundle deploy`. No new
Terraform resources to write. This provides a path for CI/CD Automation, while remaining flexible for developement.
